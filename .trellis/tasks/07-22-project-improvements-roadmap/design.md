# 技术设计：路线图产出方法

## 目标边界

本任务**只产出审查报告与实施路线图**，不修改业务代码。任何后续功能落地必须拆成独立 Trellis 子任务。

## 分析方法

1. 以代码与测试为证据，不以设想优先。
2. 按用户价值 × 风险 × 依赖排序，不按“看起来高级”排序。
3. 每个条目必须能独立验收：问题证据、方案边界、不做事项、验证命令/测试矩阵。
4. 保留现有正确边界：loopback、唯一主人、fail-closed、审批超时拒绝、不改投其他 Pi、secret 不入日志。

## 产出结构

| 文档 | 内容 |
|------|------|
| `prd.md` | 审查目标、范围、验收标准 |
| `research/*.md` | 可靠性/安全与 UX 专题证据 |
| `roadmap.md` | 面向决策的 P0/P1/P2 路线图与首个 MVP 建议 |
| `implement.md` | 本任务完成步骤与后续拆分子任务方式 |

## 优先级定义

- **P0**：会直接导致错误执行、丢审批、重复发卡、协议崩溃或身份断裂。
- **P1**：显著影响安全边界、运维诊断或核心遥控体验。
- **P2**：工程治理、测试补强、长期可维护性。

## 依赖图（实施时）

```text
协议运行时校验
  → notify requestId 幂等
    → Bridge 待确认队列 / notify_ack
      → 重连复用 piId + 审批结果 ack
        → 审批/绑定轻量持久化
审批卡片按钮（可与协议校验并行，但依赖审批状态机）
/lark status / 状态卡片（可独立）
队列查看取消（依赖 queue item id）
通知历史/重试（依赖 notify 状态机）
HTTP token/限流/脱敏（可与 P0 并行，但建议在控制面扩展前完成）
端到端测试与发布检查（全程穿插，P0 完成后加厚）
```

## 首个推荐 MVP

**审批卡片按钮 + 幂等回调**，或若更偏可靠性则先做 **协议运行时校验**。二者均可独立验收；推荐产品优先前者、工程优先后者。
