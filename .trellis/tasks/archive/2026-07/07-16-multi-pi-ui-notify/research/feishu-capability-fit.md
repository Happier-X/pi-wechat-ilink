# 飞书能否支撑 Multi-Pi 通知与回传

## 用户需求

1. 多个 Pi 实例同时运行
2. 审批 / 需要回复 / 任务结束时提醒到手机
3. 在手机上继续：批准/拒绝、回复、发下一条指令
4. 回传落到正确的 Pi（路由）
5. 通道不必是微信

## 飞书能力对照

| 需求 | 飞书能力 | 匹配度 |
|------|----------|--------|
| 推送到手机 | 机器人 IM 消息、加急（应用内/短信/电话） | 高 |
| 审批 | 交互卡片按钮（批准/拒绝）+ `card.action.trigger` | 高（比纯文本「批准 ID」更好） |
| 需要回复 | 文本消息 / 卡片输入 / 话题回复 | 高 |
| 任务结束提醒 | 机器人主动发消息 | 高 |
| 继续发指令 | 用户给机器人发消息 → 事件订阅/长连接消费 | 高 |
| 多 Pi 路由 | 消息/卡片里带 `piId`/`cwd`/`session`；或每 Pi 独立会话/话题 | 高（需自建路由层） |
| 多实例不抢同一轮询游标 | 飞书是「事件推到你的接收端」，不是多客户端抢同一 iLink cursor | 高于微信 iLink |

## 推荐架构（概念）

```
Pi A / Pi B / Pi C
    │  注册 + 上报事件（审批/需回复/结束）
    ▼
本机或轻量协调服务（路由表：piId → 进程）
    │
    ▼
飞书机器人（发卡片/消息；收消息与卡片回调）
    │
    ▼
手机飞书
```

- **出站**：某 Pi 需要审批 → 协调层发飞书卡片（含 pi 标签 + approval id）
- **入站**：用户点按钮或发文字 → 协调层按 id/默认绑定路由回对应 Pi

## 相对微信 iLink 的优势

- 交互卡片审批，不必解析「批准 ABC123」
- 开放 API / 事件模型更适合多会话元数据
- 不依赖多进程抢同一 `cursor.json`
- 可加急，人离开电脑也更容易看到

## 代价与约束

- 需要飞书企业/应用：机器人、权限（发消息、订阅事件、卡片回调等）
- 多 Pi 路由仍要**自研**（飞书不会自动知道哪个 Pi）
- 本机多 Pi 需要：本地 IPC/HTTP 注册，或每 Pi 独立轻量 webhook 端口 + 统一 bot 后端
- 与现有 `pi-wechat-ilink` 是平行通道，不必替换才能开干；也可先做飞书 MVP

## 结论

**能实现。** 飞书适合作为 Multi-Pi「提醒 + 审批 + 继续对话」的主通道；关键工程在**多 Pi 注册与路由**，不在飞书能不能推送/收消息。
